# Godot Addon Manager — Design

**Date:** 2026-05-16
**Status:** Approved

## Summary

`gam` is a command-line addon manager for Godot projects. A project checks in an
`addons.toml` manifest describing every addon it depends on and where to obtain
it. `gam` resolves and installs those addons into the project's `addons/`
directory. It behaves like a familiar package manager (npm/cargo style),
produces a committed lockfile for reproducible installs, and is fully usable
non-interactively so AI agents can drive it.

## Goals

- Declarative `addons.toml` manifest, hand-editable or edited via a TUI.
- Reproducible installs via a generated, committed `addons.lock`.
- Support three source types: git repositories, GitHub releases, and plain
  archive URLs (zip/tarball).
- Support private sources by reusing the system's existing git authentication
  and environment-provided tokens.
- First-class non-interactive use: every command works via flags, supports a
  `--json` output mode, and returns meaningful exit codes.
- An interactive TUI wizard for `add`, to make manual addon entry easy.

## Non-Goals

- Hosting or publishing addons; `gam` only consumes them.
- A central registry. Sources are addressed directly by URL/repo.
- Dependency resolution between addons (no transitive dependencies). Each addon
  entry is independent.
- Managing the user's `.gitignore`. Whether installed files are committed or
  ignored is left entirely to the user.

## Technology

- Go 1.26.
- [Cobra](https://github.com/spf13/cobra) for the CLI.
- [Bubble Tea](https://github.com/charmbracelet/bubbletea) for the TUI.
- A TOML library (`github.com/BurntSushi/toml`) for manifest parsing.
- [testify](https://github.com/stretchr/testify) for unit tests.
- Module path: `github.com/CafecitoGames/godot-addon-manager`. Binary: `gam`.

## Fetching Strategy

`gam` shells out to the system `git` binary for git sources, uses `net/http`
for archive downloads, and uses the GitHub REST API for release assets.

Rationale: private-repo auth reuses the system's existing git configuration
(SSH keys, credential helpers) at no implementation cost. The alternative, a
pure-Go git implementation, would require reimplementing SSH-agent and
credential-helper handling and would work against this decision.

Requirement: `git` must be available on `PATH`. `gam` checks for it and emits a
clear error if absent.

## Manifest: `addons.toml`

Hand-edited or TUI-edited. Declares intent, not resolved state.

```toml
[addons.dialogue_manager]
source      = "git"                               # git | github-release | archive
url         = "https://github.com/foo/dialogue.git"
version     = "v2.1.0"                             # git ref: tag, branch, or SHA
source_path = "addons/dialogue_manager"            # subdir to extract; auto-detected if omitted
install_as  = "dialogue_manager"                   # dir name under addons/; defaults to the table key

[addons.some_plugin]
source      = "github-release"
repo        = "owner/some_plugin"                  # GitHub owner/repo
version     = "1.4.0"                              # release tag
asset       = "some_plugin.zip"                    # release asset name; glob allowed
source_path = "addons/some_plugin"

[addons.raw_thing]
source = "archive"
url    = "https://example.com/thing-1.0.zip"       # zip or tarball
```

### Field reference

| Field         | Applies to        | Required | Notes |
|---------------|-------------------|----------|-------|
| `source`      | all               | yes      | `git`, `github-release`, or `archive` |
| `url`         | git, archive      | yes      | clone URL / archive URL |
| `repo`        | github-release    | yes      | `owner/repo` |
| `version`     | git, github-release | yes    | git ref / release tag |
| `asset`       | github-release    | no       | asset name or glob; required if a release has more than one asset |
| `source_path` | all               | no       | directory within the fetched tree to install; see auto-detection below |
| `install_as`  | all               | no       | install directory name under `addons/`; defaults to the table key |

### `source_path` auto-detection

When `source_path` is omitted, `gam` inspects the fetched tree:

1. If a top-level `addons/<name>/` directory exists, that single addon
   directory is used.
2. Otherwise the root of the fetched tree is used.

If detection is ambiguous (e.g. multiple directories under `addons/`), `gam`
fails with an error instructing the user to set `source_path` explicitly.

## Lockfile: `addons.lock`

Generated by `gam`, intended to be committed. Pins exactly what was installed
so other machines and future installs are reproducible.

Per addon it records:

- The resolved git commit SHA (git sources) or the downloaded asset/archive
  SHA-256 (github-release and archive sources).
- The resolved `version` and `source_path` actually used.
- A hash of the corresponding `addons.toml` entry, to detect manifest drift.

Behavior:

- `install` uses `addons.lock` when it is present and consistent with
  `addons.toml`. If an entry in the manifest has no lock entry, or the manifest
  entry's hash differs from the lock, that addon is re-resolved and the lock is
  updated.
- `update` ignores existing pins, re-resolves every (or named) addon, and
  rewrites `addons.lock`.

## Project Discovery

`gam` locates the Godot project by walking up from the current directory until
it finds `project.godot`. `addons.toml` and `addons.lock` live alongside
`project.godot`; addons install into `<project>/addons/`. If no project root is
found, `gam` fails with a clear error (except for `init`, which creates the
manifest in the current directory).

## Authentication

- **git sources:** no `gam`-managed credentials. The system `git` binary
  handles SSH keys and credential helpers as configured.
- **github-release sources:** a token read from `GITHUB_TOKEN` (falling back to
  `GH_TOKEN`) is sent on GitHub API requests when present; public repos work
  without it.
- **archive sources:** plain HTTP(S). If the URL host requires auth, the user
  embeds credentials in the URL or relies on a public URL. (No credential store
  in v1.)

`gam` never reads or writes a credential file.

## Commands

All commands accept a global `--json` flag and `-v/--verbose` /
`-q/--quiet` logging flags.

| Command           | Description |
|-------------------|-------------|
| `gam init`        | Create a starter `addons.toml` in the current directory. |
| `gam add`         | Add an addon. With no flags, launches the TUI wizard. With flags (`--source`, `--url`/`--repo`, `--version`, etc.) adds non-interactively. Writes to `addons.toml` and installs it. |
| `gam remove <name>` | Remove an addon from `addons.toml` and `addons.lock` and delete its installed directory. |
| `gam install`     | Install all addons per `addons.toml`, honoring `addons.lock`. |
| `gam update [name...]` | Re-resolve all or named addons, install, and rewrite `addons.lock`. |
| `gam list`        | List configured addons with their resolved/installed state. |
| `gam completion`  | Generate shell completion scripts (Cobra built-in). |

## Output and Exit Codes

Default output is human-readable text. With `--json`, `add`, `install`,
`update`, and `list` emit structured JSON; failures emit a JSON object with an
`error` field.

| Code | Meaning |
|------|---------|
| 0    | Success |
| 1    | Generic / unexpected error |
| 2    | Usage error (bad flags/arguments) |
| 3    | Manifest or lockfile error (parse, validation) |
| 4    | Fetch error (network, auth, source not found) |
| 5    | Install error (filesystem, extraction) |

## Package Layout

```
cmd/gam/                 main entry point
internal/cli/            cobra command definitions
internal/manifest/       addons.toml + addons.lock parse/write/validate
internal/source/         Fetcher interface + git/githubrelease/archive impls
internal/project/        Godot project-root and addons/ dir discovery
internal/installer/      source_path resolution, copy into addons/<install_as>/
internal/tui/            Bubble Tea wizard for `add`
internal/output/         text vs JSON rendering, exit-code mapping
```

### Core interface

```go
// Fetcher retrieves an addon source into a local temporary directory and
// reports the resolved version metadata for the lockfile.
type Fetcher interface {
    // Fetch downloads the source described by spec into a temp directory
    // and returns the local path plus resolved metadata.
    Fetch(ctx context.Context, spec AddonSpec) (FetchResult, error)
}

type FetchResult struct {
    Dir             string // local path to the fetched tree
    ResolvedVersion string // commit SHA or release tag actually obtained
    Checksum        string // SHA-256 of the archive/asset; empty for git
}
```

The installer is source-agnostic: it takes a `FetchResult`, resolves
`source_path` within `FetchResult.Dir`, and performs a clean replace of
`addons/<install_as>/`.

## Data Flow

1. Command loads and validates `addons.toml` (and `addons.lock` if present).
2. For each target addon, `gam` decides whether the lock pin is usable or the
   addon must be re-resolved.
3. The appropriate `Fetcher` fetches the source into a temp directory.
4. The installer resolves `source_path` and copies it into
   `addons/<install_as>/`, replacing any existing directory atomically where
   possible.
5. `gam` writes/updates `addons.lock`.
6. `output` renders the result as text or JSON and sets the exit code.

## Error Handling

- Each layer returns typed errors that map to the exit codes above.
- Fetches use a temp directory; on failure the temp directory is cleaned up and
  the existing installed addon is left untouched.
- Manifest validation runs before any network access, so configuration errors
  surface immediately.
- A missing `git` binary is detected up front with an actionable message.

## Testing

- testify for assertions across all packages.
- `manifest`: round-trip parse/write tests for `addons.toml` and `addons.lock`,
  plus validation-failure cases.
- `source`: `Fetcher` is an interface, enabling fakes. The `archive` fetcher is
  tested against `httptest` servers; the `git` fetcher against a local on-disk
  repository; the `githubrelease` fetcher against a mock GitHub API server.
- `installer`: tested with fake `FetchResult`s covering `source_path`
  auto-detection, explicit `source_path`, and clean-replace behavior.
- `cli`: command tests using fakes for the source and installer layers,
  asserting exit codes and `--json` output.
